# contributing_ai.md

> 本文件约束开发协作 AI 如何在 vimember 仓库中工作。
> 若与 `AGENTS.md` 冲突，以 `AGENTS.md` 为准。

## 1. 默认工作协议

每次接到任务后，AI 必须先判断用户要的是：

1. 只讨论或制定计划。
2. 只修改文档。
3. 读取 Figma 并做 UI 实现。
4. 编写或修改 App 代码。
5. 验证、审查或修复问题。

用户明确说“不用立刻开始制作”时，只能做文档、整理、分析和计划，不得开始写 App 功能代码。

## 2. 任务输出结构

除非用户只是问一个很小的问题，否则每个开发任务都应包含：

1. **Goal**：一句话说明本次任务目标。
2. **Known Requirements**：列出已明确的用户要求。
3. **Open Questions**：列出不能猜的待确认点。
4. **Plan**：分步计划，保证可以小步回滚。
5. **Files to Add/Change**：列出会新增或修改的文件。
6. **Verification**：说明如何验证，以及哪些未验证。
7. **Final Output**：交付总结，包含具体文件路径。
8. **Self-check**：逐项标注 `PASS`、`BLOCKED` 或 `NOT VERIFIED`。

## 3. 不猜测规则

vimember 的协作最高优先级之一是：不懂就问，不猜。

以下情况必须向用户确认：

1. Figma 未给出的视觉细节。
2. 用户没有说明的页面状态、空状态、错误状态、权限状态。
3. 摘要固定字数、截断样式和具体展示规则。
4. 首页“最主要的一条视频”的具体判定方式。
5. 任何会影响用户数据安全的行为，例如删除、覆盖、迁移。
6. 任何会影响 App Store 合规性的技术选择。

已确认的基础决策不得反复改写：

1. 视频导入后复制到 App 沙盒。
2. 文字保存到 App 本地数据库。
3. 最低支持 iOS 17.0。
4. 首页同一时间只播放最主要的一条视频。
5. 删除只删除 App 本地记录和 App 沙盒视频文件，任何时候不删除系统相册原视频。

## 4. Figma 协作规则

正式制作界面时：

1. 必须先让用户指定要读取的 Figma 页面、Frame、组件或节点。
2. 读取后再制定实现计划。
3. 实现后应尽量与 Figma 在布局、尺寸、颜色、字体、间距、圆角、阴影和状态上接近。
4. 如果 Figma 与文档冲突，先询问用户哪个为准。
5. 不得凭旧项目参考文件或通用模板自行创造页面。
6. 首页实现前必须读取 Figma `首页` Frame 和 `首页-切换` Frame。
7. 首页 `首页` Frame 内有上下超出屏幕的内容和屏幕范围标识，实现时必须适配不同尺寸 iPhone。
8. 首页中与屏幕边缘相关的悬浮按钮、边距和位置必须用真实设备屏幕尺寸按 iOS point 计算；以 Figma `iPhone Air` 屏幕范围 `420pt * 912pt` 为参考缩放，优先使用 `UIScreen.main.bounds.size`，不得用滚动内容高度或普通容器高度替代真实屏幕高度。
9. 当前已确认的首页悬浮控件基准值：右上切换按钮 `44pt * 44pt`，距右 `20pt`、距上 `69pt`；底部搜索框 `306pt * 50pt`，右下添加按钮 `50pt * 50pt`，间距 `10pt`，底部整体左右边距 `27pt`，距真实屏幕底部 `27pt`。
10. 用户准备的首页 demo 路径为 `/Users/zhangsiyuan/Documents/视频`，只能作为“文稿/视频”展示效果参考，不得参考或照搬 demo 的实现方式。
11. 如果 demo 与 Figma 存在差异，最终视觉以 Figma 为准。

## 5. 文件与变更规则

1. `指导文件参考/` 是只读参考区，除非用户要求更新参考资料，否则不要改动。
2. 新项目指导文件位于项目根目录和 `doc/`。
3. 每次修改文档后，检查是否仍有旧项目名称、旧路径或旧功能语义残留。
4. 代码实现阶段必须保持变更范围小，避免把多个主题混在同一次任务里。
5. 新增三方依赖前必须说明用途、体积影响、替代方案和是否必要。

## 6. iOS 变更原则

1. 用户相册访问必须走 iOS 公开 API。
2. 视频播放、缩略图生成、资源加载必须考虑性能和内存。
3. 长列表滚动与视频播放状态必须解耦，避免滚动时卡顿。
4. 权限被拒绝、受限授权、视频读取失败等状态必须有明确处理。
5. 任何涉及本地数据删除或视频文件删除的功能必须二次确认。
6. 删除流程不得调用删除系统相册原视频的 API。
7. 首页视频下方纯色底、渐变模糊透明融合和动态取色不得阻塞滚动主路径。
8. 首页正文超过两行时只显示两行，详情页仍展示全文。
9. 首页日期显示方式切换规则未确认前，不得自行实现具体规则。

## 7. GitHub 工作规则

1. 项目使用 GitHub 做版本管理。
2. `main` 分支只放稳定版本。
3. 开发分支默认使用 `codex/` 前缀。
4. 提交前必须说明变更范围和验证状态。
5. 未经用户明确要求，不创建仓库、不推送、不打开 PR。

## 8. 验证门槛

代码阶段至少完成以下之一：

1. Xcode Build 成功。
2. 命令行 `xcodebuild` 成功。
3. 对关键 ViewModel 或纯逻辑补充单元测试并通过。

如果当前阶段还没有 Xcode 项目或无法执行构建，必须在最终输出中标注 `NOT VERIFIED`，不能假装已验证。

## 9. 文档阶段验收

当前文档整理阶段的验收标准：

1. 参考文件已被复制到 `指导文件参考/`。
2. 新项目指导文件不再沿用旧项目的产品名称和核心功能。
3. 新文档明确写入 vimember 的视频日记需求。
4. 新文档明确写入 Figma 读取规则。
5. 新文档明确写入“不能猜，要询问”的协作规则。
6. 新文档明确标注待确认事项。
7. 新文档明确写入已确认的视频保存、数据库、iOS 版本、删除和 GitHub 规则。
