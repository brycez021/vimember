# vimember 架构规范要求

> 本文件约束 vimember 的架构演进。当前仍处于文档和计划阶段，已确认的产品决策必须进入后续实现，未确认的 Figma 与交互细节不得猜测。

## 1. 总体原则

1. 视频日记优先：核心数据单元是“视频 + 文字”。
2. 本地优先：当前需求未包含账号、云同步或后端服务。
3. Figma 优先：正式 UI 必须读取用户指定的 Figma 内容后再实现。
4. 不猜测：影响体验、数据安全、架构或视觉的未知点必须先问用户。
5. 小步可回滚：每次改动只处理一个明确目标。
6. 最低支持 iOS 17.0。

## 2. 核心模块边界

后续实现建议按以下职责划分：

1. `App`：应用入口、权限描述、根导航。
2. `Features/Import`：从系统相册导入视频。
3. `Features/Editor`：给视频添加或修改文字。
4. `Features/Home`：首页纵向视频日记列表。
5. `Features/Gallery`：画廊浏览。
6. `Features/Detail`：完整视频和文字详情。
7. `Domain`：视频日记模型、摘要规则、播放策略。
8. `Services`：相册访问、视频资源、缩略图、持久化。
9. `DesignSystem`：来自 Figma 的设计 Token 和组件。

实际目录可在创建 Xcode 工程时根据项目模板调整，但职责边界不可混乱。

## 3. 数据与持久化要求

已确认：

1. 视频导入后复制到 App 沙盒。
2. 文字保存到 App 本地数据库。
3. 本地数据库优先使用 SwiftData；如工程条件不适配，必须说明原因再更换方案。
4. 数据库记录保存本地视频路径、文字、摘要相关数据、创建/更新时间和原相册来源标识。
5. 摘要展示正文前面固定字数的一部分；具体字数、行数和截断样式后续确认。
6. 删除视频日记时，只删除 App 本地数据库记录和 App 沙盒视频文件。
7. 删除流程任何时候不得删除系统相册原视频。

必须满足：

1. 数据模型能稳定表示一条视频日记。
2. 数据读取失败时有明确降级状态。
3. 删除记录前必须二次确认。
4. 数据迁移策略在模型变化时必须可追踪。
5. 为第二版可能加入的视频分类能力保留可迁移空间，但第一版不实现分类功能。

## 4. 相册与权限

1. 导入视频必须使用 iOS 公开 API。
2. 必须处理授权、拒绝、受限授权、用户取消和资源读取失败。
3. 不得使用私有 API 读取或写入系统相册元数据。
4. 权限说明文案必须和实际用途一致。
5. 导入后必须把视频复制到 App 沙盒，再进入编辑页。

## 5. 视频播放要求

### 首页

1. 滚动过程中视频不动。
2. 滚动停止后，当前屏幕中最主要的一条可见视频立即开始播放。
3. 同一时间只播放一条首页视频。
4. 离开屏幕的视频必须及时暂停并释放不必要资源。
5. 播放状态不能阻塞滚动流畅性。
6. “最主要的一条视频”的具体判定方式需要在实现前结合 Figma 和用户确认。

### 详情页

1. 视频进入详情后自动播放。
2. 视频播放结束后循环播放。
3. 支持再次编辑文字。
4. 支持删除视频日记，但只删除 App 本地数据。
5. 退出详情页时必须正确暂停或释放播放器。
6. 后台、锁屏、来电、App 切换等生命周期需要明确处理。

## 6. UI 与设计系统

1. 任何页面实现前必须读取用户指定的 Figma 节点。
2. 颜色、字体、间距、圆角、阴影、图标、布局比例必须来自 Figma 或用户确认。
3. 不得继承旧项目参考文件中的颜色和布局。
4. 首页、画廊、详情、导入、编辑的空状态和错误状态如果 Figma 未指定，必须向用户询问。

## 7. 性能要求

1. 视频缩略图生成不得阻塞主线程。
2. 视频列表滚动必须保持流畅。
3. 播放器数量必须受控，避免多个视频同时占用大量资源。
4. 大视频文件读取必须考虑内存峰值。
5. 不得在滚动路径中执行高频日志、重计算或磁盘写入。

## 8. 可测试性要求

1. 摘要规则应设计为可单元测试的纯逻辑。
2. 数据存储服务应可替换，以便测试。
3. 播放策略应尽量与具体 View 解耦。
4. 权限和失败状态应可模拟。

## 9. 禁止事项

1. 未经确认实现账号、云同步、社交分享、滤镜、剪辑、配乐等扩展功能。
2. 未读取 Figma 就实现正式 UI。
3. 删除用户系统相册中的原视频。
4. 使用私有 API。
5. 把旧项目 SimpleMemo 的规则当成 vimember 规则。
6. 未经用户要求提前实现第一版之外的分类 UI。
