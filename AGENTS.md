# vimember Agent Guide

> 本文件定义 vimember 的产品方向、协作规则和不可随意改动的行为边界，面向 Codex / Claude / Cursor 等 AI 编程工具。
> 若与 `contributing_ai.md` 或其他文档冲突，以本文件为准。

## 1. 项目定位

**产品名：** vimember

**一句话定位：** 一个 iOS 视频日记 App，用户从系统相册导入视频，为视频配上一段文字，然后用首页列表、画廊和详情页回看这些视频记忆。

**当前阶段：** 已完成基础指导文件与关键产品决策整理，尚未开始制作 App 功能。

## 2. 参考资料边界

项目内 `指导文件参考/` 文件夹保存的是另一个 iOS 项目的指导性文件，仅作为文档结构、协作方式、风险意识的参考。

使用参考资料时必须遵守：

1. 不得照搬旧项目的产品语义、页面结构或代码路径。
2. 不得把旧项目的“照片记忆”“回忆/相册”“瀑布流”等规则当成 vimember 的规则。
3. vimember 的实际规则以本文件、用户最新要求和用户指定读取的 Figma 内容为准。
4. 如果发现旧参考文件与 vimember 要求冲突，必须忽略旧参考文件。

## 3. 核心产品要求

### 3.1 视频导入与编辑

1. 用户可以从 iOS 系统相册导入视频。
2. 用户可以给导入的视频添加文字。
3. 添加文字后完成编辑，形成一条视频日记。
4. 导入视频后必须进入编辑页。
5. 支持后续再次编辑文字；编辑入口按钮的位置和样式由用户后续通过 Figma 指定。
6. 当前需求只描述“视频 + 文字”的配对，不包含滤镜、剪辑、变速、配乐、贴纸等视频编辑能力，除非用户后续明确要求。

### 3.2 首页视图

1. 首页展示所有已加入的视频日记。
2. 展示方式为上下排列。
3. 每条内容必须把视频和它搭配的文字放在一起。
4. 首页只展示文字摘要，不展示全文。
5. 点击首页中的内容可以进入详情视图。
6. 用户上下滑动首页时，视频不动；一旦用户停止滑动，当前屏幕上最主要的一条可见视频需要立刻开始播放。
7. 首页同一时间只播放最主要的一条视频，不同时播放多条视频。

### 3.3 画廊视图

1. 画廊视图展示已经加入的视频日记。
2. 画廊视图只展示文字摘要，不展示全文。
3. 点击画廊中的内容可以进入详情视图。
4. 画廊的具体布局、尺寸、列数、封面截帧方式和交互动效必须以后续 Figma 指定内容为准，不得猜测。

### 3.4 详情视图

1. 详情视图展示完整视频和完整文字。
2. 详情视图中的视频需要自动播放。
3. 详情视图中的视频需要反复循环播放。
4. 详情视图支持再次编辑文字，具体入口由用户后续通过 Figma 指定。
5. 详情视图支持删除视频日记，具体入口、确认方式和视觉样式由用户后续通过 Figma 指定。

## 4. 已确认的产品与技术决策

1. 视频导入后复制到 App 本地沙盒，App 展示和播放使用自己的本地视频文件。
2. App 可以保存原相册视频的来源标识作为记录或未来关联，但不能依赖系统相册原视频作为唯一播放来源。
3. 文字保存在 App 本地数据库。
4. 摘要规则为展示正文前面固定字数的一部分；具体字数、行数和截断样式后续由用户确认。
5. 首页停止滚动后只播放当前屏幕中最主要的一条视频；“最主要”的判定细节在实现前结合 Figma 和用户确认。
6. 导入视频后必须进入编辑页。
7. 支持再次编辑，入口按钮后续由 Figma 指定。
8. 支持删除视频日记；删除只删除 App 本地数据库记录和 App 沙盒里的视频文件，任何时候都不删除系统相册原视频。
9. iCloud、账号、同步、备份、导出等能力后续再说，MVP 不提前实现。
10. 最低支持版本定为 iOS 17.0。
11. 第二版可能加入视频分类功能；第一版不实现分类 UI 和分类流程，但数据结构设计可以避免明显阻碍后续扩展。
12. 项目使用 GitHub 做版本管理，后续需要创建 GitHub 仓库、推送代码并做好分支管理。

## 5. Figma 使用规则

用户提供的 Figma 文件是正式 UI 实现的视觉源：

https://www.figma.com/design/z9ewZ9itwDRhTMK94G08u1/vimember?node-id=0-1&t=xXw4CqlJl4XHgkaf-1

正式制作 UI 时必须遵守：

1. 必须按用户当次指令读取用户指定的 Figma 页面、Frame、组件或节点。
2. 不得在未读取指定 Figma 内容前凭想象实现界面。
3. 不得把当前文档中的文字描述替代 Figma 视觉细节。
4. 颜色、字体、间距、圆角、阴影、图标、布局比例、状态样式和文案层级都应尽量贴近 Figma。
5. 如果 Figma 中存在不清楚、缺失或互相冲突的内容，必须先询问用户。

## 6. 不明确事项处理规则

用户已经明确要求：任何细节不懂都不要猜，而是向用户询问。

因此 AI 在开发 vimember 时必须：

1. 遇到产品行为不明确，先问用户。
2. 遇到 Figma 未指定的视觉细节，先问用户。
3. 遇到未确认的权限、播放策略、摘要规则等会影响长期架构的选择，先列出问题并等待确认。
4. 不得用“通用 App 常见做法”自动填补关键需求。
5. 可以在文档中标注“待确认”，但不得把待确认事项当成已确定规则实现。

## 7. 当前待确认问题

以下问题在正式开发前需要用户确认：

1. 摘要固定字数、行数和截断样式。
2. 首页“最主要的一条视频”的视觉和算法判定方式。
3. 画廊视图的布局：单列、多列、网格、横向、按日期分组等，由 Figma 指定。
4. 再次编辑按钮的位置、样式和触发方式，由 Figma 指定。
5. 删除入口、确认弹窗或确认流程，由 Figma 指定。
6. 首页、画廊、详情、导入和编辑页面的具体 Figma 节点。
7. 视频是否静音自动播放，以及是否显示播放控制条。
8. 是否需要浅色和深色模式。

## 8. 技术方向约束

在用户未提出其他要求前，vimember 应按 iOS 原生 App 的方向规划：

1. 使用 Swift / SwiftUI 作为默认实现方向。
2. 视频导入必须使用 iOS 公开 API，不能依赖私有接口。
3. 视频播放必须使用 iOS 公开的播放能力，并明确管理播放生命周期。
4. 最低支持 iOS 17.0。
5. 本地数据库优先使用 SwiftData；如实际工程条件不适配，必须说明原因再改用 Core Data 或 SQLite。
6. 视频文件复制到 App 沙盒后由 App 管理，数据库记录保存本地视频路径、文字、摘要相关数据和原相册来源标识。
7. 所有 UI 更新必须在主线程安全范围内完成。
8. 视频缩略图生成、文件访问、数据库读写等重任务不得阻塞主线程。

## 9. GitHub 与分支管理

1. 项目使用 GitHub 做版本管理。
2. `main` 分支用于稳定版本。
3. 开发工作默认使用 `codex/` 前缀创建功能分支。
4. 功能完成后通过 PR 合并到 `main`。
5. 未经用户确认，不直接把未验证代码推到 `main`。
6. 后续创建 GitHub 仓库、首次推送和 PR 流程需要按用户明确指令执行。

## 10. 核心行为不可破坏

以下行为属于 vimember 的核心体验：

1. 视频日记必须由“视频 + 文字”组成。
2. 首页和画廊都只展示摘要。
3. 详情页必须展示全文。
4. 首页滚动时视频不播放或不运动；滚动停止后，只播放最主要的一条可见视频。
5. 详情页视频自动循环播放。
6. 正式 UI 必须以用户指定读取的 Figma 内容为准。
7. 删除视频日记时，任何时候都不能删除系统相册原视频。

## 11. AI 工作规则

AI 修改 vimember 时必须：

1. 每次只处理用户指定的任务。
2. 不在用户只要求计划或文档时开始制作 App 功能。
3. 不做大规模无确认重写。
4. 修改前先确认当前仓库状态和相关文件。
5. 修改后说明新增或修改了哪些文件。
6. 无法验证的内容必须标注 `NOT VERIFIED`。
7. 涉及架构选择或用户体验分歧时先询问用户。
